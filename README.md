# 烛照 Zhuzhao

> 本地优先的 AI 监督助手，帮用户记录、整理、追问、复盘，并在必要时把人从拖延里骂醒。

## 项目定位

烛照是一款**本地优先（local-first）**的桌面端 AI 个人助手：

- **SQLite 是唯一主数据源**：所有 events / tasks / journal / ideas / reviews / reminders / app_logs 均存储在本地 SQLite 数据库
- **Markdown 只是导出视图**：方便用 Obsidian / VS Code 查看，但不做反向同步
- **不做云同步**：用户数据不离开本机
- **不做双向 Obsidian 同步**：不监听 Obsidian vault 变更
- **不上传用户数据**：无遥测、无账号、无联网上报
- **API Key 只保存在本地**：加密存储在 SQLite 的 `llm_providers` 表，永不出现在 Markdown 导出 / JSON 备份 / 日志中

## 配置 LLM Provider

1. 启动 App 后进入 **Settings → LLM Provider**
2. 点击"新增 Provider"，填写：
   - name（显示名）
   - provider_type（`openai` / `custom`）
   - base_url（OpenAI-compatible 网关）
   - api_key（**始终 masked 显示**）
   - model
   - temperature / max_tokens
3. 点击"测试连接"验证可用
4. 设为 active 后即可使用 Intake / Daily Summary

**安全说明**：
- API Key 只保存在本地 SQLite 的 `llm_providers` 表
- 不要把 API Key 写进 git / .env / 任何代码文件
- `.gitignore` 已忽略 `.env` / `zhuzhao.db` / `logs/`
- JSON 备份和 Markdown 导出**不会**包含 API Key

## Markdown / Obsidian 导出

### 配置

进入 **Settings → Markdown / Obsidian**：
- 启用开关
- 选择导出目录（用户自选，不能是系统敏感目录）
- 自动导出 / 手动导出
- 打开导出目录

### 导出结构

```text
Zhuzhao/
├── Daily/
│   ├── 2026-07-05.md
│   └── 2026-07-05.zhuzhao.md   (旁路文件，当用户文件冲突时生成)
├── Journal/
│   └── 2026-07-05.md
├── Tasks/
│   └── tasks.md
├── Ideas/
│   └── ideas.md
├── Backups/
│   └── 2026-07-05_1602-zhuzhao-backup.json
└── Index.md
```

### 关键规则

- **SQLite 仍是主数据源**：Markdown 只是只读视图
- **不反向同步**：用户在 Obsidian 中修改 Markdown 不会回流到 SQLite
- **不覆盖用户文件**：没有 `Generated by Zhuzhao` 标记的文件不会被覆盖
- **冲突时生成旁路文件**：当用户文件无 Generated 标记时，烛照输出到 `.zhuzhao.md` 旁路文件
- **路径安全**：所有写入通过 Rust 端 `write_export_text_file` 校验，必须位于用户保存的 exportDir 之下

## 数据与安全

### 不会提交到 git 的内容

`.gitignore` 已忽略：
- `node_modules/` / `dist/` / `*.local`
- `src-tauri/target/` / `src-tauri/gen/schemas/`
- `.trae/`
- `logs/` / `*.log`
- `.env` / `.env.local`
- `Thumbs.db`

### 数据库文件

`zhuzhao.db` 位于应用数据目录，**不提交到 git**。

### JSON 备份脱敏

Settings → 数据与安全 → "导出 JSON 备份" 会生成包含全量业务表的 JSON 文件，但：
- `llm_providers[].api_key` **强制为 null**
- `llm_providers[].base_url` **强制为 null**
- `app_settings` 中过滤敏感 key
- 不包含 Authorization / Bearer / sk- / secret / token / password

### 文件 I/O 安全

Phase 9 已删除 5 个危险 invoke command（`write_text_file` / `read_text_file` / `list_dir` / `path_exists` / `path_is_dir`），仅保留 5 个带 baseDir 校验的安全命令：

| 命令 | 安全校验 |
|---|---|
| `write_export_text_file(base_dir, path, content)` | canonicalize + is_sensitive_path + is_path_within |
| `read_export_text_file(base_dir, path)` | canonicalize + is_sensitive_path + is_path_within |
| `path_exists_in_dir(base_dir, path)` | 同上（失败返回 false） |
| `is_path_in_export_dir(base_dir, path)` | 同上 |
| `open_directory(path)` | canonicalize + is_sensitive_path |

拒绝的敏感目录：`C:\Windows` / `C:\Program Files` / `C:\ProgramData` / `C:\` / `D:\` 等系统目录。

## 已知风险 / Known Risks

详见 [docs/V0_ACCEPTANCE.md](docs/V0_ACCEPTANCE.md) 第十三节：

1. **非法目录拒绝手测暂缓**：`C:\Windows` / `C:\` / `D:\` 作为 exportDir 的拒绝已做代码级验证（Rust 端 `is_sensitive_path` + `is_path_within` 双重校验），但手动 UI 测试未完成
2. **日志诊断 UI 交互手测暂缓**：info / warn / error 筛选切换、清空日志二次确认、导出日志 JSON 等交互未手测
3. **JSON 备份仅导出不导入**：V0 只支持导出，不支持恢复
4. **Phase 5/6/7 回归手测暂缓**：数据库基线正常，但 `#phase9regression` / `#phase9reminder` 等新输入未手测
5. **ErrorBoundary 触发手测暂缓**：实际渲染异常捕获未手测
6. **不支持移动端 / 云同步 / 系统通知 / Obsidian 插件**：明确不在 V0 范围
